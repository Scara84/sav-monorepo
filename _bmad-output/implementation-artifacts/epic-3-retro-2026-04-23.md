## Analyse profonde des stories

### Patterns récurrents (positifs)

**Architecture library-style des handlers** : chaque story 3.2/3.4/3.5/3.6/3.7 pose ses handlers dans `client/api/_lib/sav/*` et les branche dans le dispatcher `api/sav.ts`. Réutilisable, testable à l'unité avec mocks Supabase chainable, permet de rester sous le cap Vercel Hobby 12 fn. Pattern à reconduire en Epic 4+.

**RPC atomique SECURITY DEFINER** : les 5 RPCs livrées (`transition_sav_status`, `assign_sav`, `update_sav_line`, `update_sav_tags`, `duplicate_sav`) suivent le même pattern qu'Epic 2.2 `capture_sav_from_webhook` — CAS sur version + audit trigger + exception mapping `P0001|code=payload`. Très cohérent, facile à durcir (F50 actor check appliqué en migration additive).

**Verrou optimiste via `version` + `WHERE version = p_expected`** : pattern zéro-conflit prouvé (tests 409 VERSION_CONFLICT verts). F61 (GET DIAGNOSTICS) le renforce contre triggers concurrents.

**Audit trail automatique via trigger Epic 1** : aucune story Epic 3 n'appelle `recordAudit()` explicitement, tout passe par le `trg_audit_*` AFTER UPDATE/INSERT qui lit `app.actor_operator_id` via `set_config`. Économie massive de code boilerplate.

**Tests mocks Supabase Proxy-chainable** : pattern Epic 2 réutilisé avec succès. Les 21 tests de `list.spec.ts` capturent les appels `.eq/.in/.gte/.lte/.ilike/.is/.contains/.textSearch/.or/.order/.limit` pour assertions, sans DB réelle.

### Patterns récurrents (problématiques)

**Sous-livraison documentée de 3.6 et 3.7** : les stories ont été marquées `review` avec Completion Notes listant explicitement les AC non livrés. Intelligent (honnêteté), mais crée une dette visible : le split V1/V1.1 acté en CR (Option C) est la résolution propre. Leçon : quand un scope story devient trop large en mi-implémentation, stopper et re-split plutôt que livrer une version réduite sans re-negotier.

**Sub-components inlinés vs spec 4 composants AC #2** : récurrent sur 3.3 (`SavStatusFilter`/`SavListTable`/`SavListFilters`/`ActiveFilterChips`) et 3.4 (`SavDetailHeader`/`SavLinesTable`/`SavFilesGallery`/`SavCommentsThread`/`SavAuditTrail`). Tasks `[x]` mensongers ont été détectés par l'Acceptance Auditor. Option C a validé le YAGNI V1 mais les specs restent à amender pour éviter le piège « déviation documentée mais task checked ».

**Schema `sav_lines` drift** : spec 3.4/3.6/3.7 cite les noms PRD-target (`unit_requested`/`unit_invoiced`/`qty_invoiced`/`credit_coefficient` numeric) mais `20260422130000_sav_schema_prd_target` aligne seulement `sav`, pas `sav_lines`. Le code consommateur utilise les noms legacy Story 2.1 (`unit`/`qty_billed`/`credit_coefficient_bp`). Décision D2 = amender Dev Notes + aligner Epic 4. Leçon : un spec cross-story doit être écrit contre la migration effectivement livrée (Story 2.1), pas contre l'idéal PRD non encore materialisé.

**Tests AC-coverage incomplète** : plusieurs stories ont livré moins de tests que les AC le demandaient (3.4 : 6/10 + 6/8 ; 3.5 : 5/7 + 10/14 ; 3.6 : 8/12 ; 3.7 : 6/8 + 3/6). Detection par Acceptance Auditor. Tous comblés en post-CR. Leçon : un compteur `tests_written / tests_specified` dans le Dev Agent Record rendrait le gap visible avant le passage en review.

### Breakthroughs

**Migration préalable `sav_schema_prd_target`** (Story 3.2 Task 0) — décision cruciale : aligner `sav` au PRD target avant d'écrire l'endpoint liste. Sans elle, les colonnes `tags`, `assigned_to`, `received_at` manquaient. Pattern : une story qui dépend d'un schéma plus riche que 2.1 doit livrer sa propre migration préalable ou dépendre d'une story d'alignement schema. À généraliser pour Epic 4.

**Helper `immutable_array_join_space`** (même migration) : workaround pour inclure `tags` dans la tsvector `search` GENERATED ALWAYS — `array_to_string` est STABLE, on a dû wrapper en IMMUTABLE. Flagué par Blind Hunter (F15) comme antipattern mais accepté V1 documenté.

**Router catch-all flat + rewrites Vercel** : contrainte Hobby 12 fn → solution élégante `api/sav.ts` + 8 `rewrites`. Non documentée dans les story artifacts (F91 CR) — à intégrer dans `docs/architecture-client.md` en Epic 4.

**Pattern CR adversarial 3 couches** : reconduit d'Epic 1/2 avec succès. 95 findings en ~10 min d'exécution parallèle. Les 3 angles (Blind Hunter adversaire, Edge Case Hunter mécaniste, Acceptance Auditor spec-oriented) sont réellement complémentaires — F36/F37 (PII leak) vient du Blind, F17 (stale fetch race) vient de l'Edge, F32 (schema drift) vient de l'Auditor. À garder comme standard de fin d'Epic.
